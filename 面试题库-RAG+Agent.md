# AI 工程面试题库（RAG + Agent）

> 来源：CSDN / 博客园 / 知乎 2025 高频面试题整理
> 用途：B 路线补课参考 + 模拟面试准备

## 一、RAG 基础（必须能答）

### Q1: RAG 系统的主要组件及工作原理？
**检索器 + 生成器**。检索器从外部知识库搜索相关信息，生成器（LLM）结合检索结果生成回答。

### Q2: RAG vs 纯 LLM 的优势？
- 减少幻觉（答案基于真实数据）
- 知识可更新（改知识库即可，不用重训模型）
- 可溯源（答案有出处）
- 适合专业领域

### Q3: 常见的检索方法？
- **稀疏检索**：BM25 / TF-IDF（关键词匹配，快但语义弱）
- **密集检索**：DPR / BERT / ColBERT（向量语义匹配，准但慢）
- **混合检索**：稀疏 + 密集结合（推荐）

### Q4: 向量数据库的作用？
存储文本的 Embedding 向量，支持快速语义相似度检索。常见：FAISS / Chroma / Qdrant / pgvector

### Q5: RAG 评估指标？
- 检索器：精确度（Precision）、召回率（Recall）
- 生成器：BLEU、ROUGE
- 整体：F1、准确率、人工评分

## 二、RAG 中级（需要理解原理）

### Q6: 怎么选检索器？
- 复杂语义查询 → 密集检索（DPR/ColBERT）
- 简单关键词匹配 → 稀疏检索（BM25）
- 综合 → 混合检索（先 BM25 粗筛，再密集重排）

### Q7: 什么是混合搜索（Hybrid Search）？
先用 BM25 快速找到候选文档，再用 BERT 等密集模型按语义重排。兼顾速度和准确性。

### Q8: 一定要向量数据库吗？
不一定。替代方案：
- 传统数据库 + 全文搜索（Elasticsearch）
- 倒排索引
- 小规模可用文件系统
取决于数据规模和语义理解需求。

### Q9: 如何确保检索信息的相关性和准确性？
1. 高质量知识库
2. 微调检索器
3. **重排序（Reranking）**— 检索后二次排序
4. 用户反馈循环
5. 持续评估（Precision/Recall）

### Q10: 处理长文档/大知识库的技巧？
1. **分块（Chunking）**
2. **摘要**
3. **分层检索**（先粗后细）
4. 内存高效 Embedding
5. 索引分片 + 并行处理

## 三、RAG 高级（面试加分项）

### Q11: 分块（Chunking）策略及优劣？
| 策略 | 优点 | 缺点 |
|------|------|------|
| 固定长度 | 简单 | 可能切断语义 |
| 按句子 | 保持句子完整 | 块太碎 |
| 按段落 | 保持上下文 | 段落可能太长 |
| 语义分块 | 上下文最好 | 实现复杂 |
| 滑动窗口 | 不遗漏信息 | 计算贵，有重复 |

### Q12: 大块 vs 小块的权衡？
- **小块**：语义集中，但丢失长距离依赖
- **大块**：上下文丰富，但信息可能被稀释
- 实践：通常 256-512 tokens，根据场景调整

### Q13: 什么是后期分块（Late Chunking）？
先对整个文档做 Embedding（token 级），再对 token 向量做分块池化。保留了完整文档上下文，质量更高但计算更贵。

### Q14: RAG 中的"情境化"（Contextualization）？
确保检索到的信息与查询高度相关。方法：
- 用 LLM 检查检索文档的相关性（如 CRAG）
- 过滤不相关文档后再送入生成器

### Q15: 先进的 RAG 变体？
- **Adaptive RAG**：根据查询动态决定是否检索
- **Agentic RAG**：引入检索代理，LLM 自主决定是否需要额外信息
- **CRAG（Corrective RAG）**：自我校正，审查检索文档相关性
- **Self-RAG**：同时评估检索文档和生成结果

### Q16: 如何减少实时 RAG 延迟？
- 缓存热门查询
- 预获取常用信息
- 流式输出
- 优化索引结构
- 模型量化/蒸馏

## 四、Agent 面试题

### Q17: 什么是 AI Agent？
具有自主决策能力的 AI 系统。核心组件：
- **感知**（接收输入/环境信息）
- **推理**（LLM 分析决策）
- **行动**（调用工具执行任务）
- **记忆**（短期对话 + 长期存储）

### Q18: ReAct 框架是什么？
Reasoning + Acting 的循环：
1. LLM 思考（Thought）
2. 决定行动（Action）
3. 获取观察（Observation）
4. 重复直到完成

### Q19: Agent 的记忆类型？
- **短期记忆**：当前对话上下文
- **长期记忆**：向量数据库存储的历史信息
- **工作记忆**：当前任务的中间状态（scratchpad）

### Q20: 多 Agent 协作模式？
- **串行**：A 做完交给 B
- **并行**：A 和 B 同时做不同子任务
- **层级**：Manager 分配任务给 Workers
- **辩论**：多个 Agent 各自回答，投票或讨论取最优

### Q21: Function Calling 怎么工作？
LLM 根据 prompt 中描述的工具列表，输出结构化的工具调用请求（函数名 + 参数），系统执行后将结果返回给 LLM 继续推理。

### Q22: Agent 的挑战？
- 幻觉导致错误工具调用
- 循环不收敛（无限循环）
- 工具调用失败的错误处理
- 成本控制（多轮调用消耗大量 tokens）
- 安全性（Agent 有执行权限的风险）

## 五、Embedding 面试题

### Q23: 什么是 Embedding？
将文本映射到高维向量空间，语义相近的文本在向量空间中距离更近。

### Q24: 余弦相似度 vs 欧氏距离？
- **余弦相似度**：衡量方向相似性，不受向量长度影响（推荐用于文本）
- **欧氏距离**：衡量绝对距离，受量纲影响

### Q25: 中文 Embedding 的挑战？
- 分词粒度影响质量
- 英文模型直接用于中文效果差
- 需要专门的中文 Embedding 模型（如 BGE-zh / GTE-zh）

### Q26: Embedding 模型选择？
| 模型 | 特点 |
|------|------|
| OpenAI text-embedding-3-small | 通用，API 调用 |
| BGE (BAAI) | 开源，中文好 |
| GTE (Alibaba) | 开源，多语言 |
| Jina | 长文本支持好 |
| Cohere embed-v3 | 多语言，质量高 |

## 六、实战问题（最容易翻车）

### Q27: 设计一个企业级 RAG 系统？
1. 文档摄入管道（读取 → 清洗 → 分块 → Embedding → 存储）
2. 查询管道（问题 Embedding → 向量检索 → Rerank → Prompt 拼装 → LLM 生成）
3. 评估体系（自动评估 + 人工反馈）
4. 生产化：缓存 / 监控 / 成本控制 / 权限管理

### Q28: RAG 效果不好怎么排查？
1. 检索质量差？→ 检查 Chunking 策略 / Embedding 模型 / 向量库配置
2. 检索到了但回答不对？→ 检查 Prompt 模板 / Context 长度
3. 完全检索不到？→ 检查文档是否被正确索引 / 查询改写

### Q29: 如何处理多语言 RAG？
- 使用多语言 Embedding 模型
- 查询时翻译 + 原文双路检索
- 分语言建立独立索引

### Q30: Token 成本怎么控制？
- 压缩 Context（只送最相关的 Top-K）
- 缓存高频查询
- 小模型处理简单问题，大模型处理复杂问题
- 监控 Token 用量，设预算告警
