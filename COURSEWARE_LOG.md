### 课件设计评估
| 主题 | 结论 | 说明 |
|------|------|------|
| W1 课件目标控制 | 保留 | 主问题聚焦，适合作为 AI 工程入门第一周 |
| W2 实验设计 | 保留 | “只改 chunking 变量”的实验设计很强，便于建立因果直觉 |
| W3 课件概念密度 | 收敛 | API、ANN、distance、metadata 同时推进，容易跑通代码但概念没吃透 |
| W4 里程碑设计 | 保留并补强 | 端到端 RAG 串联很成功，但应补失败模式和真实文档闭环 |

| W3/W4 的公共复用 | 基本合理 | 进入跨周复用阶段后，`common/` 抽象开始具备合理性 |

### 发现的课件生成经验
1. **每周只能有一个主问题**：W2 之所以强，是因为它只围绕“chunking 如何影响 retrieval”展开。主问题一旦分裂，学习者会只记步骤，不记因果。
2. **课件必须承接学习日志，而不是只承接课程大纲**：`README.md` 只能告诉你学到哪一周；真正决定“这周该怎么教”的，是 `LEARNING_LOG.md` 中暴露的知识缺陷和下次建议。
3. **实验闭环比概念罗列更重要**：好的课件不是多写背景，而是让学习者通过实验看到“现象 -> 原因 -> 结论”。
4. **代码组织必须服务教学目标**：当周核心逻辑如果过早抽到 `common/`，学习者会变成“调用已有实现”，而不是“亲手建立理解”。
5. **公共抽象要克制**：只有多周复用、语义稳定、接口边界清晰的能力，才值得进 `common/`。
6. **里程碑周必须补失败模式**：像 W4 这样的收口周，除了成功路径，还要显式写出检索错、Prompt 设计差、模型编造等失败模式。

### 常见风险
1. **只看 README 不看日志**：会让新课件只按周次推进，无法针对学习者的真实薄弱点补强。
2. **为了工程感过度抽象**：会让课件变成“项目重构练习”，而不是“概念建立练习”。
3. **只给步骤，不写预期现象**：学习者一旦结果不符合直觉，就无法判断是代码错了还是实验本来就该这样。
4. **思考题只考定义**：会让学习者看起来会了，但一到“为什么”和“工程取舍”就答不出来。

### 下次生成建议
- 生成新课件前，先读取 `README.md`、`LEARNING_LOG.md`、`COURSEWARE_LOG.md`
- 先写“本周唯一主问题”，再写实验步骤，避免先堆内容再找主题
- 先决定代码落点：哪些逻辑必须留在当周脚本，哪些能力才允许进 `common/`
- 在课件里固定补 3 块：`常见误区`、`预期现象`、`验收标准`
- 里程碑周额外补一块：`失败模式与边界`

### 课件设计评估
| 主题 | 结论 | 说明 |
|------|------|------|
| README 总路线 | 调整 | 需要明确 Agent 阶段的重点是可靠工具调用、状态管理、失败恢复，而不是只列 Function Calling / ReAct 名词 |
| W4 课件边界 | 补强 | 原先更像“Prompt + RAG 串联说明”，现在补入上下文预算和非可信上下文后更贴近真实系统边界 |
| W5 课件工程判断 | 补强 | 原先已聚焦单一主问题，但需要显式补上“召回更稳 != 回答更好”和与安全/上下文管理的连接 |
| 后续周课件生成时机 | 调整 | 不应提前全部固化，必须等学习推进后结合最新日志按需生成 |

### 发现的课件生成经验
1. **总路线和周课件必须分层管理**：`README.md` 负责方向和阶段目标，周课件负责针对当前学习状态做局部放大，不能用后者替代前者
2. **后续课件不宜提前定稿**：提前写死 W6-W13 容易与后续真实薄弱点脱节，课件应在进入对应阶段时结合最新学习日志生成
3. **安全与失败模式要前移，但仍要守住“每周一个主问题”**：可以在 W4/W5 补边界意识，但不能因此把一周课件写成多主题拼盘
4. **路线调整优先改总纲，再改临近周课件**：先改 `README.md`，再改当前即将学习的周课件，比一次性铺后续所有材料更稳

### 常见风险
1. **把“路线规划”误做成“课件预写”**：会导致内容看起来很全，但缺乏对真实学习状态的响应能力
2. **因为外部观点而过度重构路线**：容易把当前正在建立的 RAG 基础打断，造成学习主线漂移
3. **补安全边界时失去主问题聚焦**：如果一周同时塞太多主题，学习者只会记住名词，不会形成因果判断

### 下次生成建议
- 只为“即将学习的下一周”生成课件，不提前铺满后续所有周
- 生成前优先读最新一条 `LEARNING_LOG.md`，确认最近暴露的薄弱点
- 先看总路线是否变化，再决定当前周课件需不需要补边界或调整主问题
- 若只是方向调整，优先改 `README.md` 和临近周课件，不要顺手创建未来周的完整材料
