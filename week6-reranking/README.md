- 能解释为什么 `hybrid` 之后仍然需要 `reranking`

## 3. 本周核心问题

为什么召回阶段已经把候选结果找到了，最终回答却仍然不稳，而 reranking 能单独改善这个问题？

## 4. 为什么先学这个

W5 之后你已经能让候选结果更全面，但“全面”不等于“前排精准”。

如果现在不学 reranking，后面你会把很多问题都误判成：

- 混合检索还不够强
- 查询改写还不够好
- 模型生成不稳定

实际上很多时候，问题只是：

- 相关块已经在候选池里
- 但没被排进前几名

这就是 reranking 的工作范围。

## 5. 前置知识

### 必须掌握

- W5：`vector only / BM25 only / hybrid` 的差异
- Top-K 的含义，以及为什么前几名的排序质量很重要
- RAG 最终只会把少量候选块送进 Prompt

### 知道即可

- Cross-Encoder 会联合看“query + document”来打相关性分数
- Reranking 通常更慢，但精度往往更高

### 不要求现在吃透

- 所有重排序模型的训练细节
- 工业级多阶段检索系统的完整架构

## 9. 实验步骤

**第 1 步：固定候选池**
- 目的：先把问题限定在“排序”，不要同时动召回逻辑
- 操作：直接复用 W5 的 `hybrid` 结果，固定取 `Top-10` 作为候选池
- 预期现象：候选池里通常已经包含真正相关块，但顺序不总是理想
- 最小提示：不要一边调召回一边调排序，否则你很难判断收益来自哪里

**第 2 步：设计最小重排序打分方式**
- 目的：建立“重排序是在重新判断 query-document 匹配度”的直觉
- 操作：先实现一个最小版打分函数，可以是规则打分、关键词加权或模型评分接口
- 预期现象：比原始召回分数更关注“这个块是否真的回答了问题”
- 最小提示：本周重点不是追求最强模型，而是理解排序阶段的职责

**第 3 步：只重排候选，不新增候选**
- 目的：防止把 reranking 和 retrieval 混在一起
- 操作：输入固定候选列表，输出重新排序后的列表
- 预期现象：候选集合不变，但前几名的质量应该更高
- 最小提示：如果重排后出现了新块，说明你实际上又做了一次检索

**第 4 步：对比重排前后的 Top-3**
- 目的：观察 reranking 是否真的把相关块推到了前面
- 操作：打印相同查询下的“重排前 Top-3”和“重排后 Top-3”
- 预期现象：有些问题上答案块本来就在候选池中，但被重排后才进入前排
- 最小提示：重点看“位置变化”，不是只看有没有召回

**第 5 步：区分两类失败**
- 目的：训练诊断能力
- 操作：对每个查询标注属于：
  - 候选池里根本没有答案
  - 候选池有答案，但排序没排上来
- 预期现象：你会发现 reranking 只能修第二类问题
- 最小提示：这一步比“调分更高”更重要

**第 6 步：统计收益而不是只看个例**
- 目的：防止被单个成功样例误导
- 操作：准备至少 8 个问题，统计重排后命中前 3 的变化
- 预期现象：不是每题都提升，但整体前排精度应更稳定
- 最小提示：如果只看 1-2 个例子，很容易得出错误结论

**第 7 步：写工程结论**
- 目的：训练面试表达
- 操作：输出每类问题的“现象 -> 原因 -> 结论”
- 预期现象：你会开始清楚地区分召回层和排序层的职责
- 最小提示：不要写“加了 reranking 就更好”，要说明“好在哪种问题”

## 10. 预期观察表

| 场景 | 重排前现象 | 重排后预期现象 | 说明 |
|------|------------|----------------|------|
| 候选池里已有正确块 | 可能排在 4-10 名 | 更容易进 Top-3 | 这是 reranking 最擅长的场景 |
| 候选池里没有正确块 | Top-3 都不准 | 仍然无能为力 | 这是 retrieval 的问题，不是排序问题 |
| 查询语义很明确 | 前排较接近 | 细节相关性更强 | 重排序会更关注“是否真正回答问题” |
| 候选块都很像 | 排序差异不明显 | 可能收益有限 | 说明候选池本身区分度不足 |

## 11. 常见误区

- 误区：只要用了 reranking，召回问题就一起解决了  
  正解：reranking 不能凭空创造候选，只能重排已有候选

- 误区：重排序就是“再检索一次”  
  正解：重排序不负责找新块，而是负责判断“已有块里谁更该排前面”

- 误区：候选池越大越好，反正后面还能 rerank  
  正解：候选池过大意味着成本更高、噪声更多，排序模型也不是无限强

## 12. 失败模式与边界

### 失败模式 1：候选池本身没有正确答案
- 现象：重排前后都不准
- 说明：这是 W5 的召回问题，不是 W6 的排序问题

### 失败模式 2：重排序信号太弱
- 现象：重排前后顺序几乎一样
- 说明：打分函数没有真正学会区分“泛相关”和“真相关”

### 失败模式 3：重排序成本过高
- 现象：效果有提升，但延迟显著上升
- 说明：工程上要权衡精度和成本，不能只追求理论最优
