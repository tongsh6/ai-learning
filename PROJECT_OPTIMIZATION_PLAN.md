# 项目优化路线图

> 目标：把当前“按周学习脚手架”升级为“可持续迭代的 AI 实验仓库 + 面试作品底座”。

---

## 一、当前问题概览

结合现有仓库结构，当前主要问题有：

1. **工程化薄弱**
   - 依赖未统一声明
   - 缺少测试、格式化、静态检查
   - 配置项散落在脚本里

2. **代码复用性弱**
   - W1 的 embedding / similarity 逻辑需要在 W2 复制
   - 后续 Week 增多后，会出现更多重复代码

3. **实验结果不可沉淀**
   - 结果主要靠终端打印
   - 缺少统一的实验输入、输出、指标和结论记录

4. **作品化表达不足**
   - README 更像学习提纲，不像项目介绍
   - 缺少系统架构图、实验结论页、可演示入口

---

## 二、优化目标

优化后希望达到以下状态：

- 能快速运行每周实验，不依赖口头约定
- 公共能力可以复用，不需要每周复制粘贴
- 实验结果可比较、可追踪、可回顾
- 仓库结构更像真实 AI 工程项目
- 到 W10-W13 时可以自然扩展成可展示的成品

## 2026-05 校准结论

当前项目不需要推倒重来，优先做三类小幅调整：

- **实验结果前移沉淀**：从 W5 开始保存检索结果、参数、Top-K、人工判断和结论，避免到 W8 才从零建设评估体系。
- **可观测性前移**：W8-W9 开始引入最小 tracing / eval 报告；W13 负责整合展示，不再把可观测性当作最后补丁。
- **Agent 工具层补 MCP**：W10 在 Function Calling 之外增加最小 MCP 工具接口对照，重点学习工具描述、权限边界、输入校验、审计和失败恢复。

模型选择不写死具体型号。Agent 阶段只约束能力类型：低成本模型用于常规工具调用，推理模型用于复杂规划和长任务验证。

---

## 三、分阶段执行

## P0：先补地基（建议本周完成）

> 优先级最高。不做这一步，后面每一周都会继续积累技术债。

### 1. 配置外置化

**要做什么**

- 把脚本里的 API Key、URL、模型名抽到环境变量
- 新增 `.env.example`
- 统一读取方式，例如：
  - `EMBEDDING_URL`
  - `EMBEDDING_MODEL`
  - `OPENAI_API_KEY` 或其他云端 Key

**为什么**

- 避免硬编码敏感信息
- 本地 Ollama / 云端 API 切换更方便

**验收标准**

- 仓库中不再出现真实密钥
- 不改源码也能切换 embedding 服务地址和模型名

### 2. 依赖声明统一化

**要做什么**

- 新增 `requirements.txt`，至少包含当前依赖：
  - `numpy`
  - `requests`
- 如果准备长期维护，可以直接改成 `pyproject.toml`

**为什么**

- 避免 README 和真实环境不一致

**验收标准**

- 新环境只需一条命令即可安装依赖

### 3. 增加最小测试体系

**要做什么**

- 新增 `tests/`
- 先覆盖纯函数：
  - `cosine_similarity()`
  - `chunk_by_fixed_length()`
  - `chunk_by_paragraph()`
  - `chunk_by_sliding_window()`

**为什么**

- 这些函数是后续周次的基础积木
- 越早加测试，后续重构越安全

**验收标准**

- `pytest` 能跑通
- 关键边界场景有覆盖：
  - 空文本
  - 短文本
  - overlap 边界
  - 向量零模长

### 4. 增加基础代码质量工具

**要做什么**

- 加 `ruff` 做 lint
- 可选加 `ruff format` 或 `black`

**为什么**

- 后面文件增多时，统一风格比手工约定更可靠

**验收标准**

- 有明确命令可执行：
  - `pytest`
  - `ruff check .`

---

## P1：抽公共模块（建议 W2-W4 完成）

> 目标是把“每周一个脚本”升级为“每周实验 + 公共基础库”。

### 1. 拆分公共能力

**建议目录**

```text
common/
  __init__.py
  config.py
  embedding.py
  similarity.py
  io.py
  text_splitters.py
```

**职责建议**

- `config.py`：读取环境变量和默认配置
- `embedding.py`：封装 embedding API 调用
- `similarity.py`：余弦相似度、排序逻辑
- `text_splitters.py`：固定长度、段落、滑窗等分块函数
- `io.py`：保存实验结果到 `json/csv`

**为什么**

- W2 之后每周都会复用这些能力
- 后面接 Chroma、BM25、Reranker 时结构更清晰

**验收标准**

- W2 不再复制 W1 函数
- 公共逻辑只维护一份

### 2. 统一实验入口

**要做什么**

- 让每个周脚本都遵循一致模式：
  - 输入数据
  - 参数设置
  - 调用公共模块
  - 输出结果
  - 思考题

**为什么**

- 降低每周实验的认知切换成本

**验收标准**

- W1-W4 的脚本结构一致，读起来像同一个项目而不是独立小作业

### 3. 引入 CLI 参数

**要做什么**

- 用 `argparse` 支持参数化运行，例如：
  - `--model`
  - `--chunk-size`
  - `--overlap`
  - `--top-k`

**为什么**

- 方便做实验对比
- 方便后面记录“同一脚本不同参数”的结果

**验收标准**

- 不改代码也能对比不同 chunk size / overlap / model

## P2：建立实验记录、评估与最小可观测性（建议 W5-W9 完成）

> 这是从“写代码练习”走向“AI 工程方法论”的关键一步。

### 1. 结构化保存实验结果

**建议目录**

```text
experiments/
  configs/
  results/
  reports/
```

**结果文件建议**

- `json`：保存原始参数、候选结果、得分
- `csv`：保存便于横向对比的摘要结果
- `md`：保存人工总结结论

**为什么**

- 终端输出不利于长期比较
- 后面做评测时需要回看历史结果

**验收标准**

- 每次实验都能产出一份可落盘结果文件

### 2. 统一评估指标

**建议从 W2 就开始准备**

- 检索命中率
- Recall@K
- Precision@K
- MRR
- Answer Faithfulness
- Refusal Quality
- Citation Consistency
- 安全失败率

**为什么**

- W8/W9 做 RAG 评估时就不会从零开始设计

**验收标准**

- 至少有一个统一的评估模块或工具函数

### 3. 固定实验数据集

**要做什么**

- 为每周实验准备稳定输入数据
- 后续可以放到：
  - `data/`
  - `datasets/`

**为什么**

- 同一实验在不同日期、不同模型上才有可比性

**验收标准**

- 实验数据不再散落在代码常量里

### 4. 引入最小 tracing / eval 报告

**要做什么**

- 从 W8-W9 开始记录一次 RAG 请求的关键链路：
  - query
  - retrieval strategy
  - retrieved chunks
  - rerank score
  - prompt context
  - model answer
  - judge result
- 先用本地 `json/md` 报告实现，后续再接 LangSmith 或 Phoenix

**为什么**

- 2026 年的 RAG / Agent 工程更强调可诊断性
- 没有 trace，就很难判断问题出在召回、排序、上下文组装、生成还是工具调用

**验收标准**

- 任意一次失败回答都能回放关键输入、候选证据和评分过程

## P3：作品化升级（建议 W10-W13 完成）

> 目标不是“多写几个文件”，而是把仓库变成你能演示、能讲清楚、能放进面试中的项目。

### 1. 强化 README

根 README 建议补充：

- 项目目标
- 学习路线图
- 当前完成度
- 技术栈
- 如何运行
- 关键实验截图或示例输出
- 后续规划

### 2. 增加架构图

建议至少有两张图：

- `RAG 基础链路图`
- `Agentic RAG / Tool Calling 架构图`

### 3. 提供最小可演示入口

建议最终至少有一个：

- CLI Demo
- FastAPI API
- 简单 Web 界面

### 4. 整合评估与可观测性说明

在 W13 前后整合已有产物：

- 检索评估结论
- 回答质量评估结论
- LangSmith 或 Phoenix 接入说明
- Agent 工具调用 trace
- 人工审批与失败恢复记录

### 5. 补 Agent 工具层协议与安全边界

W10-W13 需要补充：

- Function Calling 与 MCP 的最小对照
- 工具 Schema、参数校验、白名单、超时、重试、降级
- 高风险工具的人类审批
- secret 不进入模型上下文
- 工具调用日志可审计
- dry-run / rollback 思路

## 四、推荐落地顺序

如果你现在就开始做，我建议按这个顺序推进：

1. 清理硬编码配置与敏感信息
2. 新增 `requirements.txt`
3. 建 `common/` 抽公共函数
4. 建 `tests/` 并补最小单测
5. 加 `ruff` 和 `pytest`
6. 给 W5/W6 实验结果落盘到 `experiments/results/`
7. 在 W8/W9 建最小 eval / tracing 报告
8. 在 W10 补 Function Calling 与 MCP 的最小对照实验
9. 到 W13 再接 FastAPI、Vue Dashboard、LangSmith 或 Phoenix

---

## 五、你这个项目最值得优先做的 3 件事

如果只做最有价值的 3 件事，我建议是：

1. **实验结果结构化落盘**
   - 当前已经进入 RAG 优化阶段，只有把参数、Top-K、判断和结论保存下来，后面评估才有可比性

2. **最小评估与 tracing 报告**
   - 这是把“感觉效果更好”升级成“能定位瓶颈”的关键

3. **Agent 阶段补 MCP 与工具安全边界**
   - 这是 2026 年 Agent 工程的关键补丁，但不需要打断当前 W5/W6 主线

## 七、一个务实判断

这个仓库现在最像“高质量学习笔记 + 实验脚手架”，还不像“AI 工程项目”。

这是正常状态，不是缺点。真正的关键不是一次性做很大，而是从现在开始：

- 让公共逻辑可复用
- 让实验结果可沉淀
- 让项目表达可展示

只要这三件事逐步做起来，到 W13 结束时，这个仓库就不只是学习记录，而会变成一个可以讲、可以跑、可以展示的作品。
