# Week 5：混合检索（Hybrid Retrieval）

> 本周唯一主问题：为什么纯向量检索在真实资料场景里不够稳定，而 `BM25 + 向量检索` 能补上这个缺口？
>
> 交付物：`05_hybrid_search.py`

## 1. 本周定位

- 本周主题：混合检索
- 在 13 周路线中的位置：第二阶段第一周，开始从“能跑通 RAG”走向“能优化 RAG”
- 承接上周什么：W4 和里程碑已经跑通 PDF 版端到端 RAG，但你已经观察到真实资料里“术语/缩写/精确名词”查询不够稳
- 为下周铺垫什么：如果 W5 只能把“召回池”变大但排序还不够准，W6 的 `reranking` 就有了清晰动机

## 2. 本周目标

- 能解释为什么纯向量检索对“精确术语”查询容易不稳定
- 能自己实现一个最小可运行的 `BM25 + 向量双路召回`
- 能比较 `vector only`、`BM25 only`、`hybrid` 三种方案的差异并说明原因
- 能用“现象 -> 原因 -> 结论”的方式表达本周实验结果
- 能理解：混合检索解决的是“召回更稳”，不是“上下文越多越好”或“系统自动更安全”

## 3. 本周核心问题

为什么纯向量检索在真实资料场景里不够稳，而混合检索能改善这个问题？

## 4. 为什么先学这个

W4 里你已经具备完整 RAG 闭环，所以现在真正的瓶颈不再是“怎么把系统串起来”，而是“为什么有些问题明明资料里有，系统却不一定找得到”。

如果这一步不先学，后面讲 `reranking`、评估指标、Query Expansion 时都会变成黑盒，因为你分不清：

- 是召回阶段就漏了
- 还是召回够了，但排序不准
- 还是检索其实没问题，回答阶段才出错

同时你也要开始建立一个工程判断：

- 检索不是召回越多越好
- 召回更多意味着上下文更长、噪声更多、潜在恶意内容也更多
- `hybrid` 的价值是让候选证据更稳，而不是免除后续排序、压缩和安全处理

## 5. 前置知识

### 必须掌握

- W2：分块策略对检索结果的影响
- W3：向量检索和 Top-K 的基本过程
- W4：RAG 管道如何把检索结果送进生成模型
- 里程碑：真实 PDF 文档中的噪声会影响检索质量

### 知道即可

- BM25 是经典关键词检索方法，关注“词是否出现、出现几次、是否稀有”
- 融合排序的目标不是“绝对最准”，而是让两路检索互补

### 不要求现在吃透

- BM25 的完整数学推导
- Reciprocal Rank Fusion、Weighted Sum 等更多融合策略
- 倒排索引和工业级搜索引擎实现细节

## 9. 实验步骤

**第 1 步：固定实验材料，减少变量**
- 目的：本周只研究 retrieval strategy，不再把变量混到 PDF 解析里
- 操作：先复用 `milestone-1/data/rag_intro_zh.txt` 作为实验文档
- 预期现象：文本材料和里程碑主题一致，但不再受 PDF 提取噪声干扰
- 最小提示：如果一周要回答两个问题，实验会失焦；本周主问题不是“PDF 清洗怎么做”

**第 2 步：保留向量检索 baseline**
- 目的：没有 baseline，就无法证明 hybrid 到底有没有价值
- 操作：继续复用 `common.search.search_chunks()` 做 `vector only`
- 预期现象：语义描述型查询通常表现还可以，但精确术语类查询可能不稳定
- 最小提示：不要一上来就删 baseline；优化的前提是有可比较对象

**第 3 步：手写最小 BM25**
- 目的：亲手建立“关键词匹配为什么能补纯向量的短板”这个直觉
- 操作：自己实现最小可用版 `tokenize`、`build_bm25_index`、`bm25_search`
- 预期现象：对于 `RAGas`、`BM25`、`Prompt Engineering` 这类词面很强的查询，BM25 往往更稳
- 最小提示：先别追求完美中文分词，先让“同词命中 > 未命中”这件事成立

**第 4 步：设计两类查询**
- 目的：让对比实验真正能说明问题
- 操作：至少准备 8 个问题，分成：
  - 精确术语类：如“RAGas 是什么？”
  - 语义描述类：如“为什么 chunk 太大会影响检索质量？”
- 预期现象：不同方案在两类问题上的强弱会明显不同
- 最小提示：如果问题全是泛泛语义问法，你很难看出 BM25 的价值

**第 5 步：实现融合排序**
- 目的：观察“两路互补”而不是“谁替代谁”
- 操作：先实现最简单的 rank-based fusion，不要一开始做复杂加权
- 预期现象：hybrid 往往不会在所有问题上都第一，但整体更稳
- 最小提示：先做稳定的简单方案，再讨论复杂公式；工程上先要可解释

**第 6 步：做对比实验**
- 目的：把“我感觉 hybrid 更好”变成“我观察到它在哪类问题上更好”
- 操作：对同一组问题跑 `vector only / BM25 only / hybrid`
- 预期现象：hybrid 对整体覆盖率更友好，尤其能减少“文档里明明有但没召回”的情况
- 最小提示：至少打印 Top-3 结果，不然你无法判断是排序问题还是完全没召回

**第 7 步：写出结论而不是只贴结果**
- 目的：训练工程表达和面试表达
- 操作：每类查询至少写一条“现象 -> 原因 -> 结论”
- 预期现象：你会开始区分“词面匹配优势”和“语义泛化优势”
- 最小提示：不要写成“hybrid 更好，因为结果更好”；要说清楚到底好在哪种问题

## 10. 预期观察表

| 场景 | `vector only` 预期现象 | `BM25 only` 预期现象 | `hybrid` 预期现象 | 说明 |
|------|------------------------|----------------------|-------------------|------|
| 查询是精确术语/缩写 | 可能漏召回或排序不稳 | 往往更稳 | 通常最稳 | BM25 擅长词面命中 |
| 查询是语义描述 | 通常较好 | 可能一般 | 通常不差 | 向量检索擅长语义近邻 |
| 查询同时包含术语和意图 | 可能只抓住语义或只抓住局部词 | 可能只抓住表面词 | 往往更均衡 | 这正是 hybrid 的价值 |
| 文档块里没有明确命中词 | 可能还能召回相近语义 | 通常较弱 | 依赖向量通道补足 | BM25 不是万能的 |

## 11. 与上下文管理和安全的连接

- `hybrid` 的目标是改善召回，不是授权你把更多结果无脑塞进 Prompt
- 召回池变大后，通常还需要 W6 的 reranking 和后续上下文压缩，否则噪声会跟着一起放大
- 如果文档里存在伪指令、误导内容或模板噪声，`hybrid` 只会更稳定地把它们召回；这不是 `hybrid` 的错，而是你后续必须做边界控制
- 工程上要始终区分三件事：`召回更稳`、`排序更准`、`送入模型的上下文更干净`

## 12. 常见误区

- 误区：混合检索就是“把两个结果拼起来”，自然就更好  
  正解：如果没有统一排序，结果只会更乱；核心是“互补后的融合”

- 误区：BM25 是老方法，所以在 RAG 里价值不大  
  正解：真实资料里大量问题都带有术语、缩写、专名，词面信号依然非常关键

- 误区：只要 hybrid 比 vector 多召回几条，就说明系统变好了  
  正解：要区分“召回更多”与“把真正相关内容排进前几名”是两回事，这正是 W6 要继续解决的问题

- 误区：hybrid 已经更稳了，所以可以少考虑 Prompt 和上下文管理  
  正解：召回更稳只解决了候选证据问题，不能替代后续的排序、压缩、拒答与安全边界设计

- 误区：既然模型能理解语义，就让模型自己先改写查询、自己决定怎么搜  
  正解：本周的重点是建立对检索机制的可解释直觉，不要把核心变量过早交给模型黑盒处理

## 13. 失败模式与边界

### 失败模式 1：术语类问题仍然没召回
- 现象：即使用了 BM25，`Top-K` 里还是没有真正相关块
- 可能原因：分块切坏了、术语被切断、tokenize 过于粗糙、查询本身就写错了

### 失败模式 2：召回到了，但前排仍然不准
- 现象：正确块出现在 `Top-10`，但 `Top-3` 里噪声更多
- 可能原因：融合排序太粗，缺少 reranking；这正是 W6 要解决的问题

### 失败模式 3：召回更多后回答反而更差
- 现象：`hybrid` 的候选池更大，但生成答案比 `vector only` 更散、更泛
- 可能原因：更多块被直接塞进上下文，噪声和冲突信息上升；这说明“召回更稳”不等于“回答更好”

### 失败模式 4：召回到了不该信任的内容
- 现象：模板残留、伪指令、无关页眉页脚、错误注释被稳定召回
- 可能原因：检索系统只是在找“相关文本”，不会自动判断它是否可信；这要求后续系统有清洗、约束和边界设计

## 14. 思考题

1. 为什么纯向量检索在“RAGas 是什么”这类查询上，可能不如 BM25 稳定？
2. 如果把查询改写成更长的自然语言描述，为什么向量检索可能又变强了？
3. 混合检索解决的是“召回不全”还是“排序不准”？为什么它不能完全替代 reranking？
4. 如果你的文档里术语很多，但同义表达也很多，你会更偏向 BM25、向量检索，还是 hybrid？为什么？
5. 如果面试官问你“为什么 RAG 系统里还要保留传统 IR 方法”，你会怎么用 1 分钟回答？
6. 为什么说 `hybrid` 改善召回，并不等于最终回答一定更好？
7. 如果召回池变大了，你会如何决定哪些块真的送进最终 Prompt？
