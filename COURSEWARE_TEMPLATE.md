# 课件生成规范模板

> 用途：以后新增 Week 课件时，先按这份规范让 AI 生成，再人工审核落地。

## 1. 这份模板要解决什么问题

当前项目的目标不是“看懂概念”，而是：

- 从全栈开发者逐步成长为能独立设计、实现、优化 `RAG + Agent` 系统的 AI 工程师
- 每周通过一个主问题建立稳定认知，而不是零散记忆一堆术语
- 通过实验形成“现象 -> 原因 -> 结论”的工程表达能力

所以，好的课件必须同时满足 3 个要求：

- 能承接上周，并为下周铺路
- 能让学习者通过实验观察到关键现象
- 能暴露误区，而不是只给结论

---

## 2. 课件质量标准

生成新课件时，必须满足以下标准：

1. 只围绕 `1 个主问题` 展开，不要做成多主题拼盘
2. 必须说明这周在整个路线中的位置：`承接什么 / 铺垫什么`
3. 生成前必须读取 `README.md`、`LEARNING_LOG.md` 和 `COURSEWARE_LOG.md`
4. 必须有实验闭环：`目标 -> 步骤 -> 现象 -> 结论 -> 验收`
5. 必须显式列出 `3 个最容易学歪的点`
6. 课件中的代码组织必须遵守当前项目架构，不能随意新起目录和模块分层
7. 必须复用前几周代码，避免突然引入新框架
8. 不直接给完整实现，只给脚手架、TODO、最小提示
9. 思考题要考“为什么”和“工程取舍”，不能只考定义
10. 必须有可面试化表达的总结模板
11. 如果课件要求抽取公共模块，必须先说明为什么不能只放在当周脚本里

---

## 3. 生成前，先给 AI 的上下文

每次让 AI 生成新课件时，先明确这些信息：

- 学习者画像：`Java/Vue 背景，Python 一般，目标是掌握 RAG + Agent`
- 当前所处阶段：`第几周`
- 上一周已掌握内容
- 上一周暴露出的知识缺陷
- 本周想建立的唯一核心能力
- 本周允许引入的新概念范围

如果项目里已有记录，AI 在生成前必须先读取：

- `README.md`
- `LEARNING_LOG.md`
- `COURSEWARE_LOG.md`

其中 `LEARNING_LOG.md` 和 `COURSEWARE_LOG.md` 都不是可选参考，而是本项目课件生成的关键输入。

至少要从 `LEARNING_LOG.md` 最近一条记录里提取：

- 上次会话覆盖了什么
- 上次暴露了哪些知识缺陷
- 下次会话建议优先补什么
- 学习者在哪类问题上容易卡住

至少要从 `COURSEWARE_LOG.md` 最近一条记录里提取：

- 上次沉淀了哪些课件生成经验
- 上次识别了哪些课件设计风险
- 下次生成课件时应优先遵守哪些规则

如果新课件没有体现这些信息，这份课件就不算合格。

此外，AI 还必须先观察当前仓库的组织方式，再决定课件里的代码落点。至少要确认：

- 公共能力是否应放进 `common/`
- 是否需要补充 `tests/`
- 是否应复用 `scripts/dev.py` 的入口方式
- 新实验是否仍放在 `weekN-xxx/` 目录下

如果课件提出的代码落点和当前仓库组织方式冲突，这份课件也不算合格。

关于是否抽到 `common/`，默认遵守下面的决策规则：

- 只被当前周使用、且主要用于教学演示的逻辑：优先放在当周脚本
- 被多周复用、且语义稳定的能力：才考虑抽到 `common/`
- 如果只是为了“看起来更工程化”而抽公共层，不应该抽
- 如果抽到 `common/`，必须在课件里说明：
  - 它会被哪些周复用
  - 不抽会造成什么重复或混乱
  - 抽取后接口边界是什么

## 4. 新课件的固定结构

以后每一周的 README 或课件文档，统一按下面结构生成。

### 4.1 本周定位

- 本周主题
- 在 13 周路线中的位置
- 承接上周什么
- 为下周铺垫什么

### 4.2 本周目标

只写 `3-4 条`，并且目标必须是“可验证的能力”，例如：

- 能解释为什么需要 reranking
- 能写出最小可运行的 hybrid retrieval
- 能对比两种方案的检索差异并给出原因

不要写这种空目标：

- 了解相关知识
- 熟悉基本概念

### 4.3 本周核心问题

只保留 `1 个核心问题`，例如：

- W2：为什么 chunking 会直接影响 retrieval 质量？
- W3：为什么需要向量数据库，而不是继续暴力搜索？
- W4：为什么 RAG 不是“检索 + 拼 prompt”这么简单？

### 4.4 为什么先学这个

必须写明：

- 它解决了前一周的什么局限
- 如果现在不学，后面哪一步会变黑盒

### 4.5 前置知识

只列本周真的会用到的前置项，不要泛泛列一堆背景。

格式建议：

- 必须掌握
- 知道即可
- 不要求现在吃透

### 4.6 实验步骤

建议 `5-8 步`，每一步都必须包含：

- 要做什么
- 为什么做
- 你将观察到什么现象
- 卡住时的最小提示

建议格式：

```markdown
**第 1 步：xxx**
- 目的：
- 操作：
- 预期现象：
- 最小提示：
```

### 4.7 预期观察表

至少给一个表，帮助学习者区分“代码错了”还是“实验现象本来就该这样”。

示例：

| 场景 | 预期现象 | 说明 |
|------|----------|------|
| A | ... | ... |
| B | ... | ... |

### 4.8 常见误区

至少列 `3 条`，并直接写出错误直觉和正确理解。

格式建议：

- 误区：以为 ...
  正解：实际上 ...

### 4.9 思考题

统一要求：

- 数量 `5 题`
- 至少 `3 题` 是“为什么/如果改条件会怎样/工程如何取舍”
- 至少 `1 题` 用来暴露边界理解
- 至少 `1 题` 用来训练面试表达

### 4.10 验收标准

本周必须明确“做到什么算学会了”。

格式建议：

- [ ] 我能独立解释 ...
- [ ] 我能自己实现 ...
- [ ] 我能比较 A 和 B 的差异并说明原因

### 4.11 与下周的连接

必须回答：

- 本周学到的东西，下周会怎么被直接复用
- 哪些问题故意留到下周再展开

## 5. 课件内容的写法要求

### 5.1 要做的

- 用工程导师口吻写，不要写成百科词条
- 优先用“问题驱动 + 实验验证”
- 对关键概念给直觉解释，但不要过度学术化
- 尽量让学习者复用已有代码，而不是重写一切
- 代码组织必须延续现有项目结构，不要为了新课件重新设计一套目录
- 抽取公共模块要克制，只有在多周复用且边界稳定时才放进 `common/`
- 引导学习者输出结论时，统一用“现象 -> 原因 -> 结论”

### 5.2 不要做的

- 不要一周塞太多核心概念
- 不要把扩展阅读放得比主线还重
- 不要只写步骤，不写“为什么这样安排”
- 不要直接给完整答案代码
- 不要脱离当前仓库结构随意新增一层 framework、utils、helpers、services 之类目录
- 不要因为“显得更高级”就把本应留在当周脚本里的逻辑提前抽象到 `common/`
- 不要让学习者在还没建立直觉时就进入复杂框架

---

## 6. AI 生成课件时必须遵守的约束

如果要让 AI 直接输出课件，请在提示词里明确要求：

- 生成前必须读取 `README.md`、`LEARNING_LOG.md` 和 `COURSEWARE_LOG.md`
- 生成前必须先观察当前项目架构，并遵守现有代码组织方式
- 不要输出完整实现代码
- 可以输出：
  - 文件结构
  - 函数签名
  - TODO 注释
  - 最小提示
- 不要跳出当前项目技术栈
- 优先复用 `common/` 和前几周实验代码
- 新课件涉及的代码文件必须优先落在现有结构中：
  - 周实验代码放在对应 `weekN-xxx/`
  - 公共能力放在 `common/`
  - 开发入口优先复用 `scripts/dev.py`
  - 测试优先补到 `tests/`
- 如果要新增 `common/` 中的公共能力，必须先解释：
  - 为什么它不是当周专用逻辑
  - 它会被哪些已有或后续周复用
  - 为什么当前就是合适的抽取时机
- 必须考虑 `LEARNING_LOG.md` 里上一周暴露的薄弱点

## 7. 给 AI 的标准提示词模板

直接复制下面这段即可：

```text
你是我的 AI 工程学习导师，请为这个项目生成第 N 周课件。

学习者画像：
- Java/Vue 背景
- Python 一般，但能在引导下完成实验
- 目标是独立设计、实现、优化 RAG + Agent 系统

生成前请先参考：
- README.md：确认整个 13 周路线和当前 Week 位置
- LEARNING_LOG.md：确认上一周暴露的知识缺陷和下次建议
- COURSEWARE_LOG.md：确认最近一次沉淀的课件生成经验和设计风险

生成要求：
1. 本周只围绕 1 个主问题展开，不要做多主题拼盘
2. 必须说明本周在整个路线中的位置：承接上周什么，为下周铺垫什么
3. 生成前必须先观察当前仓库的目录结构，并确保新课件中的代码组织遵守现有项目架构
4. 必须输出以下结构：
   - 本周定位
   - 本周目标
   - 本周核心问题
   - 为什么先学这个
   - 前置知识
   - 实验步骤（5-8 步）
   - 每一步的预期现象
   - 常见误区（至少 3 条）
   - 思考题（5 题）
   - 验收标准
   - 与下周的连接
5. 实验必须尽量复用现有项目代码，不要突然引入新框架
6. 新代码的组织方式必须遵守当前仓库架构：
   - 周实验代码放在对应 week 目录
   - 公共能力优先抽到 common/
   - 测试优先放到 tests/
   - 开发命令优先复用 scripts/dev.py
7. 如果建议把逻辑抽到 common/，必须解释为什么它不能只留在当周脚本里
8. 不要直接给完整答案代码，只给脚手架、函数骨架、TODO 和最小提示
9. 思考题要重点考“为什么”“如果改条件会怎样”“工程上如何取舍”
10. 风格要像工程导师，不要像百科
11. 额外输出两节：
   - 这周最可能学歪的地方
   - 导师应如何纠偏

输出格式：直接生成可落地的 Markdown 课件。
```

## 8. 生成后的人类审核清单

AI 生成完课件后，人工至少检查以下项目：

- 这周是否只有一个核心问题
- 是否明确参考了 `LEARNING_LOG.md` 最近一条记录
- 是否明确参考了 `COURSEWARE_LOG.md` 最近一条记录
- 是否根据学习日志里的知识缺陷做了针对性补强
- 是否根据课件日志里的经验和风险做了针对性修正
- 是否真的承接了上一周，而不是重新起炉灶
- 代码组织是否遵守了当前项目架构
- 如果抽取了 `common/`，理由是否充分
- 实验是否能在当前仓库里落地
- 有没有偷塞超出本周范围的新框架
- 思考题是否真的能区分“听懂”和“会用”
- 验收标准是否具体
- 是否照顾了上一周暴露出的知识缺陷

如果以上任一项答案是否定的，这份课件就不算合格。
